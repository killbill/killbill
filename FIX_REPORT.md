# Fix for Kill Bill issue #2127 â€” Advanced search fails on PostgreSQL when filtering by date

## Root cause

The advanced search endpoint accepts URL parameters such as
`_q=1&target_date[lte]=2025-08-28`. These are parsed by
`org.killbill.billing.util.entity.dao.SearchQuery` (`util/src/main/java/org/killbill/billing/util/entity/dao/SearchQuery.java`),
which builds a `bindMap` of values that JDBI later binds to the
JDBC statement generated by the `searchQuery` template in
`util/src/main/resources/org/killbill/billing/util/entity/dao/EntitySqlDao.sql.stg`
(see the `searchQuery(...)` and `search(...)` definitions around lines 340â€“360).

`SearchQuery` typed values from a `columnTypes` map that supported only
`Boolean`, `Integer`/`Long`/`Double`. Any other column was bound as a Java
`String`. `DefaultInvoiceDao.searchInvoices`
(`invoice/src/main/java/org/killbill/billing/invoice/dao/DefaultInvoiceDao.java`,
line 646) only declared `migrated` as `Boolean.class`. As a result
`target_date`, `invoice_date`, `created_date`, `updated_date` were all bound
as `varchar`.

MySQL silently coerces a `varchar` literal in `date <= ?` comparisons, but
PostgreSQL is strict â€” it rejects the cross-type comparison with:

```
operator does not exist: date <= character varying
```

This is the exception reported in the issue.

## Change summary

| File | Change |
| --- | --- |
| `util/src/main/java/org/killbill/billing/util/entity/dao/SearchQuery.java` | Recognise `LocalDate.class` and `DateTime.class` (Joda) in `columnTypes` and parse the raw string into a typed value via two new helpers `convertToLocalDate` / `convertToDateTime`. `convertToDateTime` also accepts an ISO date-only input (`yyyy-MM-dd`) by widening it to start-of-day. |
| `invoice/src/main/java/org/killbill/billing/invoice/dao/DefaultInvoiceDao.java` | Declare the four invoice date columns (`invoice_date`, `target_date`, `created_date`, `updated_date`) in the `columnTypes` map so they bind with the right SQL type. |
| `util/src/test/java/org/killbill/billing/util/entity/dao/TestSearchQuery.java` (new) | Pure-unit regression tests covering the new typed binding for date / datetime columns, plus assertions that the existing Boolean and untyped-String paths still work. |

`SearchQuery` lives in `util`, so any future caller (subscription, account,
payment, â€¦) can opt in to date filters by simply adding the column â†’ type entry.
Behaviour for callers that already use the API is unchanged: when no entry is
present the value continues to be bound as `String`.

## How the test exercises the fix

`TestSearchQuery` reproduces the exact URL fragment from the bug
(`_q=1&target_date[lte]=2025-08-28`), parses it through `SearchQuery` with the
new `target_date â†’ LocalDate.class` mapping, and asserts the resulting
`bindMap` entry is an instance of `org.joda.time.LocalDate` equal to
`2025-08-28`. Before the fix the bound value was the raw `String`
`"2025-08-28"`, which is what PostgreSQL was complaining about. The DateTime
tests cover the `created_date` style of column with both an ISO datetime
value and a date-only value (the latter being the friendlier UX). The
Boolean / untyped-String tests guard the existing behaviour.

The reporter's integration reproducer in `profiles/killbill` (which runs
against an embedded server with MySQL â€” and would not catch this issue on
MySQL anyway) is unaffected. Targeting `SearchQuery` directly keeps the test
fast and database-independent while precisely pinning the regression.

## Confidence: high

- The error message in the bug report names exactly the operator and types
  involved (`date <= character varying`). The fix removes the `varchar` half
  of that comparison by binding `java.sql.Date`-compatible values, which is
  the only way PostgreSQL will accept the predicate.
- The fix is localised to one helper class plus a one-line map literal in
  `DefaultInvoiceDao` â€” no SQL templates, schema or wider behaviour changed.
- The fast unit test exercises the actual bug input verbatim and pins the
  typed bind value, so the regression will be caught in CI immediately.
- Existing search paths (id / account_id / boolean migrated / balance) keep
  their previous binding because their column type entry is unchanged or
  absent.
